
Advanced FAANG Concepts in Event Driven Architecture

A. Event Replay

With Apache Kafka, consumers can replay old events.

Useful for:

rebuilding analytics
recovering services
B. Dead Letter Queue (DLQ)

Failed events go to DLQ.

OrderPlaced → Payment failed → DLQ

Prevents event loss.


C. Idempotency

Consumers may receive duplicate events.

Need safe handlers:

if (processedEvents.has(eventId)) return;

Very common FAANG interview point.

D. Eventual Consistency

Services update asynchronously.

Meaning:

Order created now
Inventory updated milliseconds later

This trades consistency for scalability.


8. Benefits
1. Scalability

Each consumer scales separately.

2. Resilience

One service failure doesn’t block others.

3. Loose Coupling

Services communicate via events.

4. Extensibility

Add new consumers without changing producer.


9. Drawbacks
A. Debugging Complexity

Hard to trace event flows.

B. Eventual Consistency

Data may be temporarily inconsistent.

C. Duplicate Handling Needed

Consumers must be idempotent.

D. Ordering Issues

Events may arrive out of order.
